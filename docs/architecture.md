## 목적
Unity/C# 기반 MVP 구현을 위한 상위 아키텍처 기준을 정의합니다.
초기 구현에서 기술 부채를 줄이고 문서-코드 일관성을 유지하는 것이 목표입니다.

## 범위
계층 구조, 책임 분리 원칙, 데이터 흐름, 기술적 제약을 다룹니다.
세부 클래스 설계는 구현 단계에서 하위 설계 문서로 확장합니다.

## 본문
### 아키텍처 상태
- 현재 코드베이스는 초기 상태이며, 본 문서는 `v0 baseline` 설계입니다.
- 미확정 항목은 명시적으로 `TBD`로 유지합니다.

### 목표 아키텍처 계층
| 계층 | 책임 | 예시 구성 |
| --- | --- | --- |
| Presentation | 입력/카메라/UI 표시 | HUD, 인벤토리 UI, 알림 |
| Simulation | 게임 규칙/틱 처리 | 채굴, 생산, 전력, 물류 계산 |
| Domain Data | 정적 데이터 정의 | ScriptableObject 기반 자원/건물/기술 데이터 |
| Infrastructure | 저장/로드/설정 | SaveData, 설정, 로깅 |

### 핵심 원칙
1. Simulation 계층은 가능한 한 Unity 프레임 의존을 줄입니다.
2. 콘텐츠 수치와 규칙은 데이터 파일 중심으로 관리합니다.
3. 기능 추가 시 문서(범위/루프/콘텐츠)부터 업데이트합니다.

### 초기 데이터 흐름 (개념)
1. 플레이어 입력 -> 배치/명령 생성
2. Simulation 틱에서 생산/소비/이동 계산
3. 계산 결과를 월드 상태와 UI에 반영
4. 체크포인트 단위로 저장 데이터 갱신(TBD)

### 저장/버전 관리 방향 (초안)
- 수정 가능한 상태 데이터는 `JSON` 기반으로 저장하고 `schemaVersion` 필드를 둡니다.
- 대용량 타임랩스 이벤트(채굴/건설/파괴 등)는 append-only 로그 구조로 분리 저장합니다.
- 이벤트 로그는 용량/성능 이슈를 고려해 바이너리 직렬화 옵션을 검토합니다.

### 비기능 요구사항 (초기)
- 단일 플레이 기준 안정적 프레임 유지
- 동일 입력 조건에서 시스템 동작 예측 가능
- 핵심 시스템(자원/생산/전력/물류)은 테스트 가능한 경계 유지

### 문서 수정 트리거
- 계층 구조나 책임 분리가 변경될 때
- 데이터 정의 방식(ScriptableObject 등)이 변경될 때
- 저장/로드 전략이 확정되거나 변경될 때

## 결정사항
- Unity/C#를 기준 기술 스택으로 사용합니다.
- 상위 계층은 Presentation/Simulation/Domain Data/Infrastructure로 분리합니다.
- 초기 설계는 v0 baseline으로 시작하고, 구현에 맞춰 점진 보완합니다.
- 저장 전략은 `JSON(상태)` + `이벤트 로그(행위)` 이원화 방향으로 진행합니다.
