## 목적
C# 코드의 가독성과 일관성을 높여 유지보수 비용을 줄입니다.

## 범위
네이밍, 파일 구성, 책임 분리, 테스트 작성의 기본 규칙을 다룹니다.

## 본문
### 네이밍 규칙
- 클래스/인터페이스/메서드/프로퍼티: `PascalCase`
- 로컬 변수/매개변수: `camelCase`
- private 필드: `_camelCase`
- 상수: `PascalCase` (도메인 의미를 드러내는 이름 사용)

### 파일 구성 규칙
- 파일명은 public 클래스명과 일치시킵니다.
- 하나의 파일에는 하나의 주요 책임(핵심 public 타입)을 둡니다.
- 네임스페이스는 기능 경로와 일치시키되 과도한 중첩을 피합니다.

### 책임 분리 규칙
- 프레젠테이션 로직과 시뮬레이션 로직을 분리합니다.
- 도메인 계산은 테스트 가능한 순수 로직으로 우선 설계합니다.
- 데이터 상수는 코드 하드코딩보다 데이터 자산으로 관리합니다.

### 테스트 권장 규칙
- 핵심 계산 로직은 단위 테스트 우선 적용
- 버그 수정 시 회귀 테스트 추가
- 테스트 이름은 `Given_When_Then` 패턴 권장

### 문서 수정 트리거
- 코딩 컨벤션이 변경될 때
- 테스트 정책이 변경될 때
- 아키텍처 책임 분리 원칙이 변경될 때

## 결정사항
- C# 네이밍/파일 구조는 본 문서를 기준으로 통일합니다.
- 테스트는 "핵심 로직 우선" 원칙으로 점진 확대합니다.
