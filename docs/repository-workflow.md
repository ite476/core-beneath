## 목적
GitHub 이슈/PR 운영에서 라벨, 마일스톤, PR 본문을 일관되게 관리하기 위한 기준을
정의합니다.

## 범위
라벨 분류 체계, 마일스톤 운영 규칙, PR 작성/리뷰 상태 운영 규칙을 다룹니다.

## 본문
### 라벨 분류 체계
| 분류 | 라벨 | 설명 |
| --- | --- | --- |
| 유형 | `유형: 문서`, `유형: 기능`, `유형: 버그`, `유형: 리팩터`, `유형: 테스트`, `유형: CI` | 작업 성격 |
| 영역 | `영역: 문서체계`, `영역: 게임플레이`, `영역: 아키텍처`, `영역: 콘텐츠`, `영역: 작업관리`, `영역: 저장소운영` | 영향 범위 |
| 상태 | `상태: 리뷰필요`, `상태: 진행중`, `상태: 차단됨`, `상태: 머지대기` | 진행 상태 |
| 우선순위 | `우선순위: P0`, `우선순위: P1`, `우선순위: P2`, `우선순위: P3` | 대응 우선도 |

### 마일스톤 체계
| 마일스톤 | 목적 |
| --- | --- |
| `Phase 0 - Documentation Baseline` | 문서 체계/운영 기준선 구축 |
| `Phase 1 - Unity Bootstrap` | Unity 초기 부팅/구조 고정 |
| `Phase 2 - Core Loop Vertical Slice` | 핵심 루프 플레이 가능 상태 |
| `Phase 3 - Progression + End Goal` | 진행 시스템/최종 목표 구현 |
| `Phase 4 - Polish + Validation` | 튜닝/안정화/마감 검증 |

### PR 작성 원칙
- PR 본문은 리뷰에 필요한 정보만 포함합니다.
- 과도한 작업 로그보다 리뷰 판단에 필요한 의사결정/범위/리스크 중심으로 작성합니다.
- 일반 구성 권장: `변경 요약`, `리뷰 포인트`, `확인 요청`.

### PR 라벨/마일스톤 적용 기준
- 기본 PR 라벨은 **최대 3개**를 권장합니다.
- 기본 조합은 `유형` 1개 + `상태` 1개 + (`영역` 또는 `우선순위`) 1개를 사용합니다.
- 초기 세팅/대규모 구조 변경 PR은 예외적으로 4개 이상을 허용합니다.
- PR은 반드시 하나의 마일스톤에 연결합니다.
- 리뷰 시작 시 `상태: 리뷰필요`, 승인 완료 시 `상태: 머지대기`로 갱신합니다.

### gh CLI 운영 예시

```bash
gh label list
gh pr edit <번호> --add-label "유형: 문서" --add-label "상태: 리뷰필요"
gh pr edit <번호> --milestone "Phase 0 - Documentation Baseline"
gh pr view <번호> --json title,labels,milestone,url
```

### 문서 수정 트리거
- 라벨 분류/명명 규칙이 바뀔 때
- 마일스톤 단계가 바뀔 때
- PR 작성 원칙이 바뀔 때

## 결정사항
- 라벨은 `유형/영역/상태/우선순위` 4축으로 운영합니다.
- PR 본문은 리뷰 중심 정보만 유지합니다.
- 일반 PR 라벨 수는 최대 3개를 기본 원칙으로 운영합니다.
- 모든 PR은 마일스톤을 필수로 설정합니다.
